## What Remains Intentionally Out of Scope

The following were reviewed but excluded from v1.2 to keep the package focused on implementation design rather than implementation execution:

- **Actual Copilot Studio agent configuration files**: Copilot Studio does not yet have a widely-used declarative configuration format suitable for version control alongside design documents. When Power Platform solution export formats stabilize, this should be added.
- **Power Automate flow definitions for Phase 2/3 tools**: Including flow definitions would require tenant-specific connector, environment, and credential configuration. These are implementation artifacts that belong in the tenant, not the design package.
- **Splunk RBAC configuration for service accounts**: The recommended service account capabilities vary by Splunk version, topology, and organizational policy. These should be defined in consultation with the Splunk service owner.
- **Benchmarking against Copilot Studio performance baselines**: The source skill's 73% improvement benchmark applies to the claude-code/Copilot CLI runtime. Equivalent Copilot Studio benchmarking requires a running tenant agent and a structured evaluation dataset.
- **Microsoft Sentinel or Defender XDR integration**: These are relevant adjacent use cases but are out of scope for the current Splunk-focused skill.

## What Still Requires Validation in a Real Tenant

None of the following can be confirmed without deploying the agent in a real Copilot Studio tenant:

1. **Instruction field behavior**: Whether the instructions produce the expected clarifying questions, refusals, and grounding behavior depends on the specific Microsoft orchestration model version active in the tenant.
2. **Knowledge retrieval quality**: Chunking strategy, source descriptions, and retrieval accuracy can only be validated against real queries and real documents in a real SharePoint knowledge store.
3. **Character limit headroom**: The paste-ready instructions are currently ~5,700 characters. The actual usable limit in Copilot Studio may vary by tenant configuration or future product changes. Validate before deployment.
4. **Channel authentication behavior**: Behavior of user-delegated vs service-account authentication in Teams vs Microsoft 365 Copilot vs custom web channel varies by tenant policy and product version.
5. **DLP policy compatibility**: Whether the agent's knowledge sources and tool actions pass Power Platform DLP policy in a given tenant depends on the organization's DLP connector classification configuration.
6. **Phase 2 tool connectivity**: Whether Power Automate can reach the Splunk REST API at port 8089 from a cloud flow depends entirely on the network configuration of the specific enterprise environment.

## Top Residual Risks

1. **Knowledge retrieval failure at launch**: If the knowledge documents are not pre-chunked, poorly titled, or not tested before the pilot, retrieval quality will disappoint early adopters and undermine trust in the agent before it has a chance to prove value. Mitigate by running all domain retrieval checks in Phase 4 before moving to Phase 5.
2. **Instruction drift without a review owner**: Agent instructions are easy to "improve" with quick edits that accidentally break safety behavior, remove version clarification rules, or weaken the Enterprise vs Cloud distinction. Mitigate by naming a single instruction owner, storing instructions in version control, and requiring a Splunk SME review for any change.
3. **Service account credential exposure**: A service account Splunk auth token stored in a Power Automate connection definition could be exposed if the environment lacks appropriate role-based access controls. Mitigate by storing credentials in Azure Key Vault, rotating on a 90-day schedule, and auditing who can view or edit the connector connection.
4. **Phantom correctness for Splunk Cloud advice**: The agent may provide confident Splunk Enterprise advice to a Splunk Cloud user if the knowledge documents do not explicitly segregate platform-specific guidance or the user does not trigger the clarifying question rule. Mitigate by creating a dedicated "Splunk Cloud vs Enterprise Differences" knowledge chunk and running V2 validation prompt at every release.
5. **No automated regression baseline**: Every instruction or knowledge update currently requires manual re-testing. Without a repeatable prompt test suite run by a Splunk SME before each change, quality degradation is invisible until a user reports a bad answer. Mitigate by formalizing the Phase 6 acceptance checklist as a standing pre-deployment gate.
